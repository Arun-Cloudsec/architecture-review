AGENT A · ARCHITECTURE REVIEW

Review

About the Review Agent How it works
Hide

What this agent does

The Review agent is a cross-domain architecture reviewer. It reads a system description and any supporting artefacts (Terraform, Kubernetes manifests, design docs, architecture diagrams, application code) and produces a prioritised list of findings across four domains — Infrastructure, Security, Application, and AI/ML — with concrete remediation and regulatory control references.

Use it for: design-stage architecture approvals, acquisition due diligence, post-incident hardening reviews, and CISO walkthroughs of new platforms. Output goes straight into an executive PDF, a technical PDF with gap analysis, or a PowerPoint for the steering committee.

Each finding carries canonical theme keys (e.g. IAM_MFA, DATA_RESIDENCY) which the Baseline agent uses to automatically compute gap analysis against any of the 18 regulatory frameworks.

Step-by-step

  1. Name the systemGive it the name you would use on a change ticket. This appears on the PDF cover and PowerPoint title slide.
  2. Describe the systemComponents, data flows, trust boundaries, regulatory scope. Be specific — 2-3 paragraphs is ideal. The agent can reason about what you tell it, not what you leave implicit.
  3. Attach artefacts (optional)Upload IaC, config, code, PDFs or architecture diagrams. PII is stripped server-side before anything leaves the host.
  4. Pick a frameworkSets the lens for the review. Well-Architected for cloud-native systems, OWASP Top 10 for web apps, CIS Controls for general hardening.
  5. Run Review30-60 seconds. You'll see an exec summary, domain scores (0-100 per domain), a severity-sorted findings table, and four export formats (Exec PDF, Tech PDF, PPTX, JSON).
  6. Drill into a baselineJump to the Baseline workspace and hit Gap Analysis from Last Review to see how this system scores against CBUAE, SAMA, HKMA, MAS, PCI-DSS or any of the other 13 baselines.

AGENT B · THREAT MODELLING

Threat Modelling

About the Threat Agent How it works
Hide

What this agent does

The Threat agent produces a STRIDE-based threat model for the system you describe, with every threat mapped to a MITRE ATT&CK for Cloud technique ID (e.g. T1078.004 for Valid Accounts: Cloud Accounts). It decomposes the system into components and data flows, walks STRIDE systematically, and proposes mitigations that address the root cause — not just compensating controls.

Use it for: design-phase threat modelling before a change goes to production, red-team scoping, security-architect hand-offs between teams, and evidence packs for regulatory submissions that require documented threat analysis.

STRIDE categories: Spoofing · Tampering · Repudiation · Information disclosure · Denial of service · Elevation of privilege. The agent labels each threat with its STRIDE letter and the affected component, so you can filter and prioritise by attack class.

Step-by-step

  1. Name the systemE.g. "Payments API v2.4" or "Customer Data Lake". Appears in the output header and exports.
  2. Describe components and trust boundariesList each component (API Gateway, Lambda, RDS, KMS, etc.) and describe where data enters/leaves trust zones. The more explicit the boundaries, the sharper the threat model.
  3. Attach artefacts (optional)Architecture diagrams are especially useful here — the agent can analyse PNG/JPEG visually and spot trust boundary crossings.
  4. Generate ModelThe agent produces a threats table with ID, STRIDE category, component, threat scenario, MITRE technique, severity, and mitigation.
  5. Filter by STRIDE category or severityUse the chip filters to focus on, say, only Information Disclosure threats or only Critical-severity threats.
  6. Export & hand overThe Technical PDF export includes the full threats table with MITRE mappings — suitable for attaching to a Change Advisory Board submission.

AGENT C · COMPLIANCE

Baseline Library & Assessment

About the Baseline Agent How it works
Hide

What this agent does

The Baseline agent is the compliance brain. It knows 18 regulatory frameworks — UAE (UAE IA, NESA, CBUAE, DESC), GCC (NCA ECC, SAMA, CBB), Asia-Pacific (HKMA CFI, MAS TRM), and International (CIS v8, NIST CSF, NIST 800-53, PCI-DSS, ISO 27001/27002, HIPAA, SOC 2, SWIFT CSP). Every control is tagged with canonical theme keys, and findings from Review / Threat / Risk carry the same tags.

That tagging is what makes the gap analysis real: a finding about "no MFA on privileged accounts" resolves automatically to CBUAE 6.1.1, SAMA 3.3.1, HKMA CFI-PR.1, MAS TRM-10, NIST 800-53 IA-2, PCI-DSS 8.3 and fifteen other controls — no keyword matching, no manual mapping.

Use it for: regulator submissions, audit evidence packs, multi-jurisdiction reporting (one review → gap analysis against every applicable framework), and internal compliance briefings that need to demonstrate control coverage.

Step-by-step

  1. (Optional) Run a Review firstIf you want gap analysis, start by running a Review, Threat or Risk assessment — the Baseline agent needs a set of findings to map against the baseline.
  2. Browse baselinesPick any baseline from the left panel. You'll see authority, version, and the full control catalogue with the canonical themes each control addresses.
  3. Run gap analysisIf you've run a review, click Gap Analysis from Last Review (top right) or the button inside the baseline detail panel. The agent resolves every finding's theme keys against every control in the chosen baseline.
  4. Read the resultControls are marked COVERED (findings touched this control and severity is medium or below), AT-RISK (findings touched the control with critical or high severity), or NOT-ASSESSED (outside the review's scope). Coverage % shows addressed ÷ total.
  5. Export for evidenceThe Technical PDF export includes the full gap analysis table tied to the baseline of your choice (?baseline=CBUAE query param in the URL).
  6. Cross-referenceThe Cross-map view (left sidebar) shows the full 70 × 18 matrix — useful for proving that a single finding impacts multiple frameworks at once.

AGENT D · RISK REGISTER

Risk Register

About the Risk Agent How it works
Hide

What this agent does

The Risk agent is the translator for the boardroom. It takes a system description and produces a business-risk register in the language your Executive Risk Committee actually uses — regulatory penalties, reputational harm, operational outage, financial loss — not CVE numbers or IAM policy excerpts.

Every risk is scored on a 5 × 5 likelihood × impact matrix, producing an inherent score (1-25 before controls) and a residual score (after the proposed controls land). Every risk gets a treatment recommendation: avoid, transfer, mitigate or accept.

Use it for: ERC/RMC submissions, board risk reports, M&A risk due-diligence, post-audit translation (turning technical audit findings into risk statements the business sponsor understands), and insurance/cyber-insurance renewal packs.

Step-by-step

  1. Name the systemBusiness-language name the ERC will recognise — e.g. "Corporate Lending Platform", not "CLP-v3.1-prod". Appears on all exports.
  2. Describe what's at stakeFocus on business context: what does the system do, who depends on it, what regulatory scope applies, what's the revenue or customer exposure. Technical detail helps — but framing matters more here than in Review.
  3. Attach supporting docs (optional)Existing risk registers, business impact analyses, or architecture diagrams give the agent richer context for scoring.
  4. Build Risk Register30-60 seconds. You'll get a risk register table with ID, business-language title, inherent → residual scores, treatment, and mapped control references.
  5. Review inherent vs residualThe difference between the two is the value of the proposed controls. A risk scored 20 → 4 tells the board the mitigation delivers 16 points of risk reduction.
  6. Cross-reference to regulationsBecause each risk carries canonical theme keys, the Baseline agent can show exactly which CBUAE, SAMA, HKMA, MAS or NIST clauses each risk maps to — essential for "regulatory risk" framing in the RCSA.
